# Feature Description Format

## Header

    # Feature: [Title]

    **Date:** YYYY-MM-DD
    **Status:** Scoped

## Recommended Sections

Adapt to the feature. Not all are required.

- **Overview**: 1-3 sentences â€” what the feature does and why.
- **End-User Capabilities**: Numbered list of what a user can do when complete. Frame as outcomes, not implementation.
- **Architecture / Scope**: High-level design, component breakdown, or scope boundaries.
- **Technical Details**: Technology choices, data models, APIs. Include rationale.
- **Risks and Considerations**: Known risks, constraints, dependencies.
- **Non-Goals / Future Iterations**: Explicitly out of scope.
- **Success Criteria**: Verifiable conditions for completion.
- **Open Questions**: Unresolved items (if any remain).

## Quality Checklist

- Scope is precise: reader can tell what is included and excluded
- End-user capabilities are concrete and testable
- Rationale is included for significant decisions
- Technical detail is appropriate to feature complexity
- No implementation plan or task breakdown (that is a separate phase)
